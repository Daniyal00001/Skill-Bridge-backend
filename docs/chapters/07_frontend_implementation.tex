\chapter{Frontend Implementation}

\section{Overview}

The \projectname{} frontend is a single-page application (SPA) built with \textbf{React 18}, \textbf{Vite 5}, and \textbf{TypeScript}. The application is structured around role-based routing, server-state management via TanStack Query, and a consistent design system built on Shadcn UI / Radix UI primitives with a Tailwind CSS utility layer.

\section{Application Entry Point}

The Vite entry point (\texttt{main.tsx}) renders the \texttt{App} component, which wraps the entire application in a provider stack:

\begin{lstlisting}[language=TypeScript, caption={App.tsx Provider Stack (Simplified)}]

          </Routes>
        </BrowserRouter>
      </TooltipProvider>
    </AuthProvider>
  </QueryClientProvider>
</ThemeProvider>
\end{lstlisting}

\section{Authentication Context}

The \texttt{AuthContext} provides global authentication state to the entire application. It exposes:

\begin{itemize}
  \item \texttt{user} --- the currently authenticated user object (or null).
  \item \texttt{token} --- the JWT access token held in memory (not localStorage).
  \item \texttt{login(credentials)} --- calls \texttt{POST /api/auth/login} and stores the response.
  \item \texttt{logout()} --- calls \texttt{POST /api/auth/logout} and clears state.
  \item \texttt{isLoading} --- true during the initial auth check on mount.
\end{itemize}

The access token is stored in JavaScript memory (not \texttt{localStorage}) to prevent XSS token theft. The refresh token is stored as an HttpOnly cookie, inaccessible to JavaScript.

\section{Route Guards}

Three components implement the route guard hierarchy:

\begin{description}
  \item[\texttt{ProtectedRoute}] Checks \texttt{AuthContext.user}. If null, redirects to \texttt{/login} with a \texttt{returnTo} query parameter.
  \item[\texttt{GuestRoute}] The inverse --- if the user is already authenticated, redirects them to their role-appropriate dashboard (\texttt{/client}, \texttt{/freelancer}, or \texttt{/admin}).
  \item[\texttt{RoleRoute}] Accepts an \texttt{allowedRole} prop (\texttt{"CLIENT"}, \texttt{"FREELANCER"}, or \texttt{"ADMIN"}). If the authenticated user's role does not match, renders a 403 Forbidden screen or redirects.
\end{description}

\section{Multi-Step Project Posting Form}

The \texttt{PostProject.tsx} page is the most complex component in the frontend, with 8 sequential steps validated independently. It uses \texttt{React Hook Form} with persistent state across steps via a parent-level form context. Draft saving calls \texttt{PATCH /api/projects/:id} with \texttt{status: "DRAFT"} on each step transition. The final ``Publish'' action sends the complete payload with \texttt{status: "OPEN"}.

\section{Freelancer Onboarding Flow}

The \texttt{Onboarding/} directory contains a step-based wizard that guides new freelancers through profile creation:

\begin{enumerate}
  \item Basic Info (name, tagline, bio)
  \item Skills (skill picker with proficiency levels)
  \item Portfolio (image + description + tech stack per item)
  \item Certificates (title, organisation, date, URL)
  \item Education (school, degree, year)
  \item Availability and Preferences (hourly rate, response time, languages)
  \item Social Links (GitHub, LinkedIn, portfolio URL)
  \item Verification (ID document upload, profile photo upload)
\end{enumerate}

A persistent progress bar shows the completion percentage fetched from the backend after each step.

\section{Design System and Theming}

The frontend implements a design system based on CSS custom properties (design tokens) with two themes: \textit{light} and \textit{dark}. The \texttt{ThemeProvider} context toggles a \texttt{data-theme} attribute on the \texttt{<html>} element, which activates the corresponding token set. Theme preference is persisted in \texttt{localStorage} under the key \texttt{"skillbridge-theme"}.

The Shadcn UI component library provides a set of accessible, headless primitives (Button, Dialog, Select, Table, Tabs, Toast, etc.) that are styled using the project's Tailwind CSS configuration. All interactive components meet WCAG 2.1 AA accessibility guidelines.
